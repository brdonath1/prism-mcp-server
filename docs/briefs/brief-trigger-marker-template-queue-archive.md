# Brief: PRISM MCP Server — Update bootstrap marker template for queue/archive split

> **Session:** S77 (04-27-26)
> **Scope:** `brdonath1/prism-mcp-server` only (no Trigger changes; no migration of existing markers)
> **Objective:** Change the bootstrap-generated `.prism/trigger.yaml` marker template so newly-enrolled projects default to `brief_dir: .prism/briefs/queue/` with `post_merge: [notify, archive]`. Pairs with the Trigger `archive` post_merge action shipped at S77 commit `37f3d1cda` — together they realize the queue/archive split design.
> **Out of scope:** Migrating existing enrolled projects' markers (separate brief — that touches per-project repos, not the server). Changing marker schema validation. Any Trigger daemon changes.
> **Deployment target:** Railway-deployed PRISM MCP server. Push to `main` triggers Railway auto-deploy. Verification is server-side via the `prism_bootstrap` tool against a fresh test scenario.
> **Compatibility:** Brief 2 is safe to deploy independently of Trigger daemon state. If a marker is dropped with `post_merge: [notify, archive]` and the daemon is running pre-Brief-1 code, the unknown `archive` action is silently skipped per existing post-merge handler logic. No coordinated restart required.

---

## Context

Brief 105 (PR #24, merged S76) added bootstrap-time `.prism/trigger.yaml` marker drops to `prism_bootstrap`. The marker template hard-codes a `brief_dir: .prism/briefs/` layout and `post_merge: [notify]` action list. This was correct at the time — there was no archive action yet.

S77 shipped the Trigger-side `archive` post_merge action (commit `37f3d1cda` on `brdonath1/trigger`). It auto-moves a brief from `<brief_dir>` to a sibling `<brief_dir>/../archive/` location after PR merge, BUT only when the brief's path contains a `/queue/` segment. Markers without `/queue/` in their `brief_dir` fall back to no-op (warn + skip).

For new projects enrolling AFTER this brief lands, the marker template needs to default to the new layout so `archive` actually runs:

- `brief_dir: .prism/briefs/queue/` (was `.prism/briefs/`)
- `post_merge: [notify, archive]` (was `[notify]`)

Existing markers on prism, prism-mcp-server (and any other already-enrolled project) are NOT modified by this brief. Brief 105's idempotency rule (do not overwrite an existing marker) is preserved — that logic should not change.

### Existing implementation (verified S77)

The marker template is generated in `src/tools/bootstrap.ts` (prism-mcp-server). The current marker text on disk for both prism and prism-mcp-server, captured this session:

```
# Trigger enrollment marker — auto-generated by prism_bootstrap.
# Presence of this file (with enabled: true) enrolls this repo in Trigger.
# Edit values below to customize per-project behavior; set enabled: false to opt out.
enabled: true
brief_dir: .prism/briefs/
brief_pattern: "brief-*.md"
branch_strategy: main-only
intra_project_parallel: false
max_parallel_briefs: 1
post_merge:
  - notify
```

This is the exact string template (or string-builder output) that needs updating in two fields.

The test surface is `tests/bootstrap-trigger-enrollment.test.ts`. Existing assertions about marker content will need updating to match the new defaults.

The feature gate `TRIGGER_AUTO_ENROLL` env var controls whether the marker is dropped at all — it does NOT need to change. Default remains `true`.

## Changes

### 1 — Update the marker template's `brief_dir` and `post_merge` defaults

**File:** `src/tools/bootstrap.ts`

Locate where the marker file content is constructed (template literal, multi-line string concatenation, or YAML object — whatever the current implementation uses; CC reads the file to determine the exact form). Apply two changes:

1. **`brief_dir`** — change the default literal value from `.prism/briefs/` to `.prism/briefs/queue/`. Trailing slash matches the existing template's style (the current value also ends in `/`).

2. **`post_merge`** — change the default action list from `[notify]` (or `["notify"]` in JS, or `- notify` in YAML output) to `[notify, archive]`. Order matters: `notify` first, then `archive`. This matches the order assumed in Trigger's post-merge handler tests (notify before archive).

Update the marker comments at the top of the file to reflect the new layout. Suggested comment block (CC may adjust wording for clarity, but must preserve the layout-explanation intent):

```
# Trigger enrollment marker — auto-generated by prism_bootstrap.
# Presence of this file (with enabled: true) enrolls this repo in Trigger.
#
# Layout:
#   brief_dir/  — pending briefs Trigger should poll and dispatch
#   archive/    — completed briefs (auto-moved by post_merge: [archive] after PR merge)
#
# Edit values below to customize per-project behavior; set enabled: false to opt out.
```

The existing comment lines that don't conflict can stay. The `archive/` reference is the only substantive new content.

### 2 — Preserve idempotency

**Same file: `src/tools/bootstrap.ts`**

Verify (do not change) that the existing logic skipping marker write when `.prism/trigger.yaml` already exists on the target repo continues to work. This is the property that protects existing markers (prism, prism-mcp-server) from being silently rewritten by this update. Brief 105 explicitly added this — it must remain in place.

If the idempotency check is implemented but seems fragile (e.g., relies on a specific GitHub API error code that could change), do NOT rewrite it as part of this brief. Flag it in the PR description as a follow-up concern but ship the template change.

### 3 — Update tests in `tests/bootstrap-trigger-enrollment.test.ts`

Existing tests that assert specific marker content strings will fail with the new defaults. Search the test file for:

- string assertions referencing `brief_dir: .prism/briefs/` (without `queue/`)
- string assertions referencing `- notify` as the sole `post_merge` entry
- inline YAML or string snapshots of the full marker

Update each to reflect the new defaults: `brief_dir: .prism/briefs/queue/` and `post_merge: [notify, archive]` (or however the test currently encodes that list).

Add at least one NEW test asserting the new marker content end-to-end — the goal is for a future regression that flips a default back to break the test. Use the style of the existing tests in that file.

Run the full test suite after all changes:

## What This Guarantees

After this brief ships and Railway redeploys:

- **New project enrollments via `prism_bootstrap` get the queue/archive default layout.** Their first brief that lands and merges will be auto-archived by the Trigger daemon (assuming the daemon is running Brief 1 code, which it is as of S77).
- **No existing project is affected.** Brief 105's idempotency continues to protect prism, prism-mcp-server, and any other already-enrolled project. Their markers retain `.prism/briefs/` and `[notify]` until Brief 3 explicitly migrates them.
- **Backward-compatible with daemon state.** A new marker with `post_merge: [notify, archive]` works correctly on a daemon running pre-Brief-1 code (`archive` silently skipped) AND on a daemon running Brief 1 code (`archive` actually runs). No coordinated deploy required.

What this brief does NOT guarantee (deliberate scope cuts):

- Existing enrolled projects don't get the new layout. Brief 3's job.
- The bootstrap doesn't pre-create the empty `.prism/briefs/queue/` directory. Git doesn't track empty dirs anyway; the directory springs into existence with the first committed brief in it. This is correct for fresh repos with no pending briefs.
- No new ntfy events. The existing `post_merge_complete` continues to be the operator-visible signal.
