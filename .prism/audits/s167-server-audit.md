## Executive summary

The audit produced **117 verified findings: 34 defects, 33 risks, 16 inefficiencies, 3 optimizations, 21 doc-drift, 10 dead-code** (16 high-severity).

**SRV-01 (chartered seed) is fully root-caused with byte-exact reproduction.** `prism_log_insight` appends ` — STANDING RULE` *after* a title whose tail already carries `[TIER:B]` (the tool has no `tier` parameter, so the operator embeds the tag in the title), minting tag-then-suffix order; brief-451's end-anchored `TRAILING_TIER_TAG` accepts only suffix-then-tag, so the tier silently defaults to A. Independently, the last rule before trailing file content swallows `## Formalized` + the EOF sentinel into its `procedure`. Running the HEAD parser on the real prism registry reproduces the observed S167 census exactly (26/92/21/139 vs expected 25/93/21/139); exactly 1 of 134 headers uses the unparseable order. Pre-451 code (unanchored tag regex) parsed it as B — PR #71 regressed this one live shape while correctly fixing mid-title false positives.

**The systemic theme: the intelligence loop fails silently at multiple points.** The PDU prompt instructs an output format its own auto-apply parser rejects 100% of the time (SRV-10); full finalization discards the entire Opus draft except handoff.md (SRV-19); three pipelines record failed GitHub pushes as success — apply-pdu archives false "applied" provenance and deletes the pending updates (SRV-02), background synthesis logs "pushed" (SRV-15), boot-test reports write-path "verified" (SRV-16); and a refusal/truncated synthesis response can overwrite intelligence-brief.md with empty content (SRV-07). Archival/rotation defects (SRV-04/05/06) silently lose the *newest* state. **Payload (D2):** 115,842 B measured; **76.1% is rules delivery** (Tier A bodies 34,281 + template 31,346 + index 19,737 + deprecated alias 2,836); project state is only ~21 KB — the oversize is a rules-delivery problem, with ~10–20 KB of low-risk diet available and the boot index currently undocumented to its only consumer (SRV-87). **Reliability:** transient 401s (INS-311) are never retried on any GitHub path (SRV-35); auth is OR-composed — a missing Bearer token is accepted whenever the IP allowlist is disabled (SRV-36). **Operations:** a `.prism/audits/`-only PR — this very deliverable — is born unmergeable under current branch protection (SRV-38).

## Dimension 1 — SRV-01: seed defect root-cause

### SRV-01 — Standing-rule tier tag silently unparsed when it precedes the `— STANDING RULE` suffix; the server's own writer mints that order; last-rule procedure swallows trailing file content

**defect / high / fix M (parser S + writer S + fixtures S)** · `src/utils/standing-rules.ts:42,48,57,77,97-120` · `src/tools/log-insight.ts:70-75,132-140`

**Observed (S167 prism boot, ground truth):** census `tier_a=26, tier_b_indexed=92, tier_c_indexed=21, total=139, from_standing_rules_file=134, from_insights=5` vs S166-expected `25/93/21/139/5`; INS-316 delivered in the Tier A bodies array with `tier:"A"`, its `procedure` carrying `## Formalized` + `<!-- EOF: standing-rules.md -->`.

**Root cause (a) — tier misclassification.** The real prism `.prism/standing-rules.md` was fetched read-only during this audit and the INS-316 header line hexdumped. It ends:

```
…do not re-add archived entries to live until config fix lands [TIER:B] — STANDING RULE
```

— **tag-then-suffix** order, clean UTF-8, no invisible characters (`e2 80 94` is the em-dash). The HEAD extractor accepts only **suffix-then-tag**:

- `standing-rules.ts:42` — `TRAILING_TIER_TAG = /(?:—\s*STANDING\s+RULE\s*)?\[TIER:([A-Z])\]\s*$/i` requires the title line to END with `[TIER:X]`, optionally *preceded* by `— STANDING RULE`. Against `[TIER:B] — STANDING RULE` it cannot match.
- `standing-rules.ts:107` — on no match, `tier` keeps its initializer `"A"`. Silent by construction: the only warn path (`:114`) fires for a *matched* tag with an unknown letter; an unmatched tag is indistinguishable from "no tag at all".
- `standing-rules.ts:57,120` — `TRAILING_TITLE_DECORATIONS` strips only the trailing `— STANDING RULE` run, so the delivered title still ends with a visible `[TIER:B]` — exactly how the operator transcribed it into the brief.
- The inconsistency is internal to one file: `INSIGHTS_RULE_SUFFIX` (`:48`) *does* accept a tag after the suffix (`— STANDING RULE [TIER:X]$`), so the two trailing-decoration grammars in the same module disagree about ordering.

**Empirical reproduction (HEAD parser via vite-node, real 394,693-byte registry):** `total=134 A=25 B=88 C=21`, `INS-316 tier="A"`. With the union's 5 insights.md rules (1×A INS-304 + 4×B) this reproduces the observed S167 census **byte-for-byte** (26/92/21/139/134/5); re-ordering the single title reproduces the S166-expected 25/93/21. Of 134 registry header lines, 117 end suffix-then-tag or bare-tag; exactly one — INS-316 — ends tag-then-suffix.

**Where the order came from — the server's own writer.** `prism_log_insight` has no `tier` or `topics` parameter (`log-insight.ts:70-75` schema), so the operator carries the tag inside `title`. The writer then appends the qualification suffix AFTER the title: `` `### ${id}: ${safeTitle}${standingTag}` `` (`log-insight.ts:136,140`, `standingTag = " — STANDING RULE"`). Any title ending in a tier tag is thereby converted to the one ordering the parser cannot read: **the system mints headers its own parser misclassifies.** This is not an operator formatting error; it is a writer/parser grammar mismatch.

**Regression vector.** Pre-brief-451, tier extraction used an *unanchored* `/\[TIER:([A-Z])\]/i` (old `standing-rules.ts:53`, `git show 1bf1194^`) which matched INS-316's tag anywhere in the line → Tier B. PR #71 (brief-451 / D-255) end-anchored the tag to kill mid-title false positives (INS-179's backticked literal; INS-308/INS-310 self-references) — correct goal, but the accepted grammar lost the one live ordering the writer produces. None of brief-451's added tests covers tag-then-suffix (`tests/standing-rules.test.ts:125-130,194-195` — every fixture is suffix-then-tag or bare-trailing-tag).

**Root cause (b) — trailing-content bleed into `procedure` (independent defect).** `extractStandingRules` splits sections only on `^### ` (`standing-rules.ts:77`), so the LAST rule's section extends to end-of-file; the procedure extractor takes *everything* after `**Standing procedure:**` (`:97-102`) with no terminator. Mid-file rules are implicitly bounded by the next `### `; only the last-rule-before-trailing-section/EOF shape leaks. Reproduced on fixtures and the real file (`procedure` tail = `"…retention shape.\n\n## Formalized\n\n<!-- EOF: standing-rules.md -->"`). Both halves of SRV-01 manifest on any registry whose final rule precedes trailing non-rule content — i.e., on every file written by the `## Formalized`-anchored insert path (`log-insight.ts:228-231`).

**Why it matters (THE WHY):** tier misclassification silently promotes Tier B bodies into every boot payload — bytes against the 100KB tripwire and delivery of rules the operator deliberately deferred to lazy-load — while the bleed injects structural noise (section headers, EOF sentinels) into rule text the model is instructed to obey. Both corrupt the context channel invisibly; no diagnostic reports either.

**Fix specification.**
1. **Parser, ordering (S):** treat the trailing decoration run as order-insensitive — match/strip `(?:\s*(?:—\s*STANDING\s+RULE|\[TIER:([A-Z])\]))+$`, extracting the tier from the tag wherever it sits in the run; emit a `STANDING_RULE_TIER_TAG_UNPARSED` diagnostic when a title tail contains `[TIER:` but extraction fails (kills the silent default-A path).
2. **Parser, bleed (S):** terminate the procedure slice at the first `^#{1,2} `-level heading or `<!-- EOF:` sentinel inside the section (equivalently: pre-strip the registry tail before splitting).
3. **Writer (S):** add optional `tier` (A|B|C) + `topics` parameters to `prism_log_insight`; compose canonical `### ID: Title — STANDING RULE [TIER:X]`; when a supplied title already ends with a tag in either order, normalize to canonical rather than appending blindly.
4. **Repro fixture (S):** registry file-tail fixture mirroring the real shape — final rule titled `… [TIER:B] — STANDING RULE`, then `## Formalized`, blank line, `<!-- EOF: standing-rules.md -->`; assert `tier==="B"` and procedure free of trailing content; plus a census-shape test asserting A/B/C counts over a multi-rule fixture using *both* decoration orders.

**Missing tests that would have caught it:** a tag-then-suffix case in `tests/standing-rules.test.ts`; an end-of-file rule fixture asserting the procedure terminates before `## Formalized`.

**Related verified findings:** SRV-11 (topics regex destroys hyphenated topic lists — same parse path), SRV-12 (B/C rules with empty topics unreachable via prism_load_rules), SRV-13 (empty-procedure delivery), and the good-news verdict that brief-453's auto-detect **fully fixed the finalize session-log archiver path** that INS-316's own rule text warns about (`finalize.ts:59-70` + `archive.ts:78-82,271-277`) — the rule's "until config fix lands" precondition is satisfied at HEAD, while the same inversion class survives un-fixed in `input-budget.ts` (SRV-04).
