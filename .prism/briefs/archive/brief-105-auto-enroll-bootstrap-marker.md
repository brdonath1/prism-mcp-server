## Context

Trigger (the autonomous brief dispatcher daemon at `brdonath1/trigger`) already has marker-file discovery shipped (S49). It enumerates all repos under `GITHUB_OWNER`, fetches `.prism/trigger.yaml` from each, validates schema, and auto-enrolls any repo where `enabled: true`. Currently only `brdonath1/prism-mcp-server` has this marker.

**Goal of this brief:** make `prism_bootstrap` drop the marker into the target project repo on first session, idempotent thereafter. Combined with a coordinated `trigger` brief running in parallel that adds periodic re-discovery, this delivers auto-enrollment: every project the operator opens a session on becomes Trigger-watched without manual config edits.

The marker schema is fixed by Trigger's `discoverMarkerProjects` validator (`brdonath1/trigger:src/config/discovery.ts`). This brief commits to that schema as ground truth — verify the validation rules from the source before authoring marker content, NOT this brief's prose summary.

## Spec

### Change 1 — Add marker-drop step to `prism_bootstrap`

In `src/tools/bootstrap.ts`, inside the existing `prism_bootstrap` handler, add a step that runs BEFORE the response is built (so it doesn't extend client wall-clock time perceptibly — sub-second when no commit needed; ~1–2s on first-time write):

1. **Check if marker exists.** Attempt to fetch `.prism/trigger.yaml` from the target project repo via the existing `fetchFile` helper. Wrap in try/catch — a 404 is the expected "marker absent" case, NOT an error.
2. **If marker present** (any successful fetch, regardless of contents): log at debug level, do nothing. The bootstrap response is unaffected.
3. **If marker absent** (404 / Not found from GitHub): construct marker content using the spec below and push to `.prism/trigger.yaml` on `main` of the target repo.
4. **Surface outcome in the bootstrap response.** Add a top-level field `trigger_enrollment` to the bootstrap JSON response with shape:
   ```ts
   trigger_enrollment: {
     status: "marker_present" | "marker_created" | "skipped" | "error",
     reason?: string,  // present when status === "skipped" or "error"
   }
   ```
   This lets future Claude.ai sessions surface a one-line note in the boot banner ("Trigger marker created" / "Trigger marker already present"). Banner integration is out of scope for this brief — just emit the field.
5. **Failures are non-fatal.** Any error during marker check or write logs at `warn` and sets `trigger_enrollment.status = "error"` with the error message in `reason`. The bootstrap response is otherwise unaffected. A failed marker write must NEVER cause the bootstrap call to fail — operators must always be able to start a session even if Trigger enrollment is broken.

### Change 2 — Marker content

The marker content for newly-bootstrapped projects:

```yaml
# Trigger enrollment marker — auto-generated by prism_bootstrap.
# Presence of this file (with enabled: true) enrolls this repo in Trigger.
# Edit values below to customize per-project behavior; set enabled: false to opt out.
enabled: true
brief_dir: .prism/briefs/
brief_pattern: "brief-*.md"
branch_strategy: main-only
intra_project_parallel: false
max_parallel_briefs: 1
post_merge:
  - notify
```

**Field choices (justified):**
- `enabled: true` — operator's stated UX is "every project I open auto-enrolls." Default opt-in. Operators can flip to `false` to disable per project.
- `brief_dir: .prism/briefs/` — PRISM convention. PF-v2's `docs/briefs/` is a historical exception; new PRISM projects should use `.prism/briefs/`.
- `brief_pattern: "brief-*.md"` — same convention used everywhere.
- `branch_strategy: main-only` — required by validator (non-empty string). Matches the existing `prism-mcp-server` marker. Means "no staging branch reset" in `post_merge`.
- `intra_project_parallel: false` and `max_parallel_briefs: 1` — safe defaults. Per-project override via marker edit.
- `post_merge: [notify]` — minimum useful action.

**Commit message for the marker drop** (note this is for the marker file commit, NOT the PR for this brief):

```
prism: enroll {slug} in Trigger via marker file

Auto-generated by prism_bootstrap on first session. Edit
`.prism/trigger.yaml` to customize behavior or opt out.
```

### Change 3 — Feature flag

Gate the entire marker-drop behavior behind a new env var `TRIGGER_AUTO_ENROLL` in `src/config.ts`:

- Default: `true` (auto-enroll enabled).
- When set to `false` (or `0`): skip the entire marker check + write, set `trigger_enrollment: { status: "skipped", reason: "TRIGGER_AUTO_ENROLL=false" }`.

This lets operators disable the feature server-side without modifying every project's marker.

### Change 4 — Tests

Add tests to `tests/bootstrap.test.ts` (or a new file `tests/bootstrap-trigger-enrollment.test.ts` if cleaner). Cases:

1. Marker absent → `pushFile` called with the canonical content; response contains `trigger_enrollment: { status: "marker_created" }`.
2. Marker present → `pushFile` NOT called; response contains `trigger_enrollment: { status: "marker_present" }`.
3. `TRIGGER_AUTO_ENROLL=false` → neither check nor write runs; response contains `trigger_enrollment: { status: "skipped", reason: ... }`.
4. `fetchFile` throws non-404 error → response contains `trigger_enrollment: { status: "error", reason: ... }`; bootstrap still returns success.
5. `pushFile` throws → same as (4) — error captured, bootstrap still returns success.

Use the existing test fixtures and mock patterns already in `tests/bootstrap.test.ts`.
