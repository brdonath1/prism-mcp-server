# Brief 439 — R8: unified drift-proof banner

**Status: PENDING**
**Repo:** prism-mcp-server
**Origin:** D-240 Phase B, audit brief-431 row R8. Independent of the archival chain;
the standalone banner deliverable flagged in the Phase B note. **CODE + DOCS.**

## Context
The boot banner (`banner_text` from `prism_bootstrap`) and the finalization banner
(`prism_finalize`) are generated by separate code paths, and a deprecated HTML
finalization widget (`banner_html`) still lingers. There is no version handshake
between the server's banner contract and the framework template that renders it
(core-template-mcp.md Rule 2 / Rule 11 Step 6), so drift goes undetected. Rule 2's
null-`banner_text` fallback also contradicts the documented single-line fallback.

## Required Changes
**Investigate first.** Read `src/tools/bootstrap.ts` (`banner_text` assembly),
`src/tools/finalize.ts` (finalization banner + any `banner_html`), and any shared
banner helper (locate the real generators — do not assume a path). Confirm the
current MCP template's expected banner spec.

1. **Single banner generator.** Factor boot + finalization banner rendering into ONE
   server-side text generator that produces the `banner_text` field for both
   surfaces. No second format. Boot and finalize are byte-consistent by construction.
2. **`banner_spec_version` handshake.** Emit a `banner_spec_version` from the server
   and compare it against the version the template declares; on mismatch, log a
   `BANNER_DRIFT` diagnostic (warn) in the response `diagnostics`. Visibility only —
   do NOT block.
3. **Deprecate the HTML finalization widget.** Stop producing `banner_html` HTML;
   keep the field null/absent for backward-compat if any consumer reads it. Document
   the deprecation.
4. **Resolve the null-fallback contradiction.** Make the server's null-`banner_text`
   behavior match the single-line fallback the template specifies (Rule 2 fallback).
5. **`docs/banner-spec.md`.** One authoritative banner contract: line-by-line boot
   banner structure, finalization banner structure, the `banner_spec_version`
   handshake, and the null fallback. This becomes the single source future banner
   changes cite.
6. **CHANGELOG.md.** Add entries for D-84 + D-85 (banner / Rule-9 history) and the MCP
   template versions 2.10.0 through 2.18.0 that were never logged.
